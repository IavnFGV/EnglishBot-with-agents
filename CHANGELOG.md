# Changelog

## 2026-04-20
- Added the thinnest Telegram admin flow for teacher-workspace workbooks: `/workbook_export <teacher_workspace_id>` now returns the generated `.xlsx`, workbook import now works by uploading a `.xlsx` document with caption `/workbook_import <teacher_workspace_id>`, student-workspace targets fail clearly, permission checks stay on teacher workspace membership, workbook success replies include compact summaries, and the new Telegram layer stays thin by delegating filename/summary work to `englishbot/workbook_admin.py`.
- Added the focused workbook import slice for teacher workspaces: `englishbot/workbook_import.py` now performs atomic workbook-key-based partial upserts from one `.xlsx`, keeps topic-item linking additive, upserts `ru`/`uk`/`bg` translations without deleting on empty cells, ignores the display-only `image` column in favor of `image_ref`, validates required sheets/columns plus duplicate workbook keys, broken refs, and invalid archive values up front, and preserves shared lexemes safely when a learning item headword changes.
- Extended the workbook export format for teacher content with a display-only `image` column next to `image_ref`; export now writes a Google Sheets-compatible `=IMAGE(...)` formula when `image_ref` is present, while `image_ref` remains the source-of-truth asset field and the new column is documented for future import to ignore as derived display data.
- Added the minimal teacher-workspace workbook export slice: `englishbot/workbook_export.py` now exports exactly one teacher workspace into a valid `.xlsx` with exactly 3 sheets (`topics`, `topic_items`, `learning_items`), workbook-key-first identity, wide `ru`/`uk`/`bg` translation columns, archive flags, asset refs, deterministic sheet/column order, and focused tests covering workbook validity, scoping, and exclusion of student-workspace runtime copies.
- Prepared teacher-workspace workbook-key persistence without adding import/export yet: `topics` and `learning_items` now store stable `workbook_key` values, startup backfills legacy rows, SQLite generates missing keys on insert, workspace-scoped uniqueness is enforced, and focused tests now cover generation, legacy backfill, uniqueness, and unchanged teacher-content flows.
- Hardened teacher topic flows to require explicit authoring workspace context: topic-based `/assign` now uses `/assign <student_user_id> <teacher_workspace_id> <topic_name>`, `/granttopic` now uses `/granttopic <student_user_id> <teacher_workspace_id> <topic_name>`, teacher topic lookup is scoped to one specified teacher workspace instead of first-match search across all teacher workspaces, duplicate topic names across workspaces are covered by focused tests, and shared student workspace resolution now fails closed when membership is ambiguous instead of silently picking the first workspace.
- Introduced the teacher/student workspace MVP: `workspaces.kind` now distinguishes authoring (`teacher`) and runtime (`student`) workspaces, invite joins create a shared student workspace instead of sharing the authoring catalog directly, teacher-only content editing helpers were added for topics and learning items, and discovery queries now hide archived content by default.
- Added the minimal publish/add flow by making assignments and topic grants copy teacher-workspace content into student workspaces under the hood, reusing published copies by source id so topic rename, topic membership updates, and translation/text fixes can safely affect future launches without adding versioning or sync frameworks.
- Hardened runtime stability for existing learning flows: topics and learning items now support archive-only lifecycle, assignments keep fixed student-workspace item ids and title snapshots, active training sessions snapshot prompt/answer data per item so live content edits do not change in-flight questions, and focused tests now cover workspace kinds, teacher-only editing, publish behavior, archive visibility, and session/assignment stability.

## 2026-04-19
- Moved assignments and student topic access onto the workspace model for the current MVP learning flows: SQLite now stores `workspace_id` on both `assignments` and `student_topic_access` with legacy backfill migrations, invite joins now also attach users to the shared `Starter Content` workspace, homework and topic-access business rules validate workspace membership instead of teacher-link ownership, `/assign`, `/granttopic`, `/topics`, and homework launch keep working in the current implicit workspace context, and focused schema/membership/visibility tests were added for the migrated flows.
- Moved learning-content ownership into workspaces for the MVP slice: `learning_items` and `topics` now store `workspace_id`, startup backfills legacy rows into a deterministic default `Starter Content` workspace, seeded starter packs live in that workspace, topic lookups/listing now resolve through workspace-scoped helpers, cross-workspace topic-to-item links are rejected in the topic module, `lexemes` remain global, and focused migration/ownership/seed tests were added while preserving current Telegram compatibility flows through the default workspace.
- Added the first workspace foundation slice: SQLite now bootstraps `workspaces` and `workspace_members`, `englishbot/workspaces.py` provides minimal create/add/list/find-by-role business logic for `teacher` and `student` memberships, current Telegram flows remain on the existing teacher-student model, and focused tests cover schema and workspace membership behavior.
- Added minimal student topic access for self-learning: SQLite now stores `student_topic_access`, teachers can grant one linked student access to one persisted topic via `/granttopic <student_user_id> <topic_name>`, students can open `/topics` to see only allowed topics and start topic-scoped training sessions, homework remains a separate assignment model, and focused tests cover schema, grant/list/check permissions, handler flow, accessible start, and inaccessible rejection.
- Added persistent topic MVP in SQLite: runtime now stores first-class `topics` plus `topic_learning_items`, starter packs (`weekdays`, `months`, `colors`) seed as real topics linked to existing `learning_items`, `/assign <student_user_id> <topic_name>` now resolves persisted topics while keeping the raw `learning_item_id` fallback, and focused tests cover topic schema, persistence, seeding, and topic-based assignment titles.
- Fixed a startup/runtime audit bug for new Telegram users: incoming and outgoing interaction logging now creates a minimal `users`/`user_profiles` stub before writing to `interactions`, so fresh updates no longer fail with `sqlite3.IntegrityError: FOREIGN KEY constraint failed`.
- Added minimal named-group homework assignment on top of the seeded starter packs: `/assign` now prefers pack names like `weekdays`, `months`, and `colors`, assignments store a human-readable title, student homework lists show that title, and the legacy raw `learning_item_id` path still works as a fallback.
- Added a minimal basic-vocabulary seeder for the current SQLite runtime: it populates 3 starter topic packs (`weekdays`, `months`, `colors`) as regular `learning_items` with `ru`, `uk`, and `bg` translations, keeps reruns idempotent, and is covered by focused tests.
- Shifted the student UX to homework-first: `/start` now shows a main `Домашка` entry when active assignments exist, otherwise a simple no-homework message, while `/learn` stays available only as a fallback command.
- Added minimal homework persistence and flow: SQLite now stores `assignments` plus ordered `assignment_items`, teachers can create fixed `learning_item` assignments for linked students via `/assign`, students can open and start assigned homework from inline buttons, and finishing an assignment marks it completed.
- Added student homework notifications for new assignments and focused tests covering assignment persistence, active-homework visibility, homework-first `/start`, homework launch from assigned content, teacher assignment handling, and notification delivery.
- Added a minimal `/learn` training session MVP over stored vocabulary: SQLite now persists `training_sessions` and ordered `training_session_items`, the bot asks one prompt at a time from translations with lexeme fallback, validates plain-text answers case-insensitively, and finishes with a short summary.
- Added focused training tests covering session creation, ordered item selection, question retrieval, correct and incorrect answer handling, session completion, and thin non-runtime handler coverage for `/learn`.
- Added minimal SQLite vocabulary persistence for `lexemes`, `learning_items`, and multilingual `learning_item_translations`, including nullable `image_ref` and `audio_ref` fields, enforced SQLite foreign keys, and focused create/read tests for schema, migration, integrity, and content readback.
- Expanded `export_ai_file_report.py` so the generated report now includes the full contents of each discovered `AGENTS.md` or agent-related file, making it usable as a one-run AI context dump.
- Added `export_ai_file_report.py`, a minimal script that scans the repository for `AGENTS.md` and agent-related files and writes their names, relative paths, full paths, and sizes into a text report file.
- Moved role storage out of Telegram `users` and into a separate `user_profiles` entity so product-level profile fields can grow independently from Telegram transport metadata.
- Relaxed invite join behavior so a `teacher` can consume their own invite code for self-testing or future self-assignment flows without losing the `teacher` role.
- Added minimal teacher-student invite binding: SQLite bootstraps `user_profiles`, `invites`, and `teacher_student_links`; `/invite` generates one-time codes for `teacher` users, and `/join <code>` binds one student to one teacher in the MVP.
- Added focused persistence and handler tests covering invite creation, successful join, invalid or reused codes, and duplicate student-link rejection, and listed `pytest` in `requirements.txt` for the new automated tests.
- Added a minimal async Telegram bot package with `/start`, `.env` token loading, INFO logging, and basic error logging for local and devcontainer runs.
- Added a root `.gitignore` for local secrets, Python cache files, virtual environments, logs, IDE artifacts, and local SQLite databases.
- Migrated the minimal Telegram runtime from `python-telegram-bot` to `aiogram 3.x` with `Dispatcher` + `Router`, long polling, and the same `/start -> Привет` behavior.
- Extended the minimal aiogram bot with one `/start` inline button, one callback handler returning `Кнопка нажата`, and one plain text echo handler replying with `Ты написал: <text>`.
- Added minimal SQLite persistence for Telegram users and regular text messages, plus a `/me` command that shows the saved user id and message count.
- Clarified `requirements.txt`: runtime depends on `aiogram` and `python-dotenv`, while SQLite uses the Python standard library and needs no separate package.
- Added a low-level SQLite interaction log with explicit `in/out` direction and event types, so incoming commands/text/callbacks and outgoing bot messages are all visible in one table.
- Moved interaction logging out of handler logic and onto aiogram-wide hooks: incoming updates are logged through `dispatcher.update.outer_middleware(...)`, and outgoing Telegram API calls are logged through bot session middleware.
- Split the aiogram audit hooks into a dedicated `englishbot/audit.py` module and moved singleton `router`/`dispatcher` setup into `englishbot/runtime.py` so `bot.py` stays focused on handlers and startup.
- Removed the separate `messages` table and switched `/me` message counting to the low-level `interactions` audit log; startup now drops the obsolete `messages` table if it exists.
