# Current State

- Repository currently contains a minimal runnable Telegram bot foundation at `englishbot/`.
- Entry point is `python -m englishbot`.
- Runtime uses long polling with `aiogram 3.x`.
- `/start` is now homework-first for students: it upserts the Telegram user into SQLite, shows a main inline button labeled `Домашка` when the student has active assignments, and otherwise replies that homework is not assigned yet while keeping `/learn` as a fallback.
- Plain text messages are echoed back as `Ты написал: <text>` and are tracked only through the low-level `interactions` audit log.
- `/me` shows the user's first name or username, the `telegram_user_id`, and the number of saved text messages.
- `/me` also shows the user's stored `role`.
- Configuration is read from `TELEGRAM_BOT_TOKEN`, with `.env` support via `python-dotenv`.
- SQLite storage uses the stdlib `sqlite3` module with a local `englishbot.sqlite3` file by default, overridable through `ENGLISHBOT_DB_PATH`.
- `requirements.txt` stays minimal but now includes both runtime packages (`aiogram`, `python-dotenv`) and the test dependency `pytest`; SQLite is not listed because it comes from the Python standard library.
- SQLite schema is created automatically on startup and now bootstraps `users`, `user_profiles`, `interactions`, `workspaces`, `workspace_members`, `invites`, `teacher_student_links`, `lexemes`, `learning_items`, `learning_item_translations`, `topics`, `topic_learning_items`, `student_topic_access`, `training_sessions`, `training_session_items`, `assignments`, and `assignment_items`; startup also creates a deterministic default content workspace named `Starter Content`, and migrations now backfill `assignments.workspace_id` and `student_topic_access.workspace_id` for legacy rows.
- Product-level user data is now separated from Telegram transport data: `users` stores Telegram identity fields, while `user_profiles` stores runtime profile fields such as `role`.
- `interactions` is the low-level audit table for Telegram traffic and stores `telegram_user_id`, `direction` (`in` or `out`), `interaction_type`, `content`, and `created_at`.
- Minimal vocabulary persistence is now present in SQLite: `lexemes` stays global, `learning_items` stores the main learning unit linked to a lexeme and now belongs to a workspace through `workspace_id`, and `learning_item_translations` stores multiple translations per learning item by language without duplicating workspace ownership.
- Repository now includes a minimal SQLite seeder for starter vocabulary packs; the current seeded packs are `weekdays` (7 items), `months` (6 items), and `colors` (6 items), and startup seeds them into the default `Starter Content` workspace as real persisted `topics` linked to workspace-owned `learning_items` with `ru`, `uk`, and `bg` translations.
- `learning_items` now supports nullable local asset references through `image_ref` and `audio_ref`, with a small startup migration that adds those columns if an older `learning_items` table already exists without them.
- Minimal learner training is now present: `/learn` still creates one active training session per user over a fixed ordered slice of up to 5 `learning_items`, but the fallback compatibility path now reads from the default content workspace instead of treating all learning items as globally usable; it asks one prompt at a time, validates plain-text answers against the lexeme headword, and ends with a short total/correct summary.
- Training persistence now uses `training_sessions` for active/completed session state and `training_session_items` for the ordered `learning_item` ids inside a session; `training_sessions.assignment_id` is nullable so the same session flow can be reused for homework.
- Training prompt selection is intentionally narrow: prefer a `ru` translation, then `uk`, then the first stored translation, and finally fall back to the lexeme headword when no translation exists.
- SQLite connections now enable foreign key enforcement, so `learning_items.lexeme_id` and `learning_item_translations.learning_item_id` are validated at runtime instead of acting as soft references.
- Vocabulary create/read persistence lives in `englishbot/vocabulary.py`; it intentionally exposes only the minimal helpers needed to create lexemes, learning items, translations, and read one learning item together with its translations.
- Homework business logic now lives in `englishbot/homework.py`; topic persistence/read helpers live in `englishbot/topics.py`; thin student homework Telegram wrappers live in `englishbot/homework_handlers.py`; `englishbot/training.py` is reused for both `/learn` and homework sessions; and `englishbot/bot.py` still only wires handlers plus `/me`, echo, and startup.
- `/me` counts saved user text messages by counting `interactions` rows where `direction='in'` and `interaction_type='text'`.
- Startup removes the old `messages` table if it exists, because it is no longer used.
- Interaction logging is attached at the aiogram framework level instead of inside specific handlers: incoming updates go through a dispatcher update middleware, and outgoing Bot API requests go through session middleware.
- Audit logging now tolerates first-contact users: before inserting into `interactions`, persistence creates a minimal `users` row plus default `user_profiles` row when that Telegram id has not been saved yet, preventing foreign-key failures on fresh updates.
- If the bot token is missing, startup fails with a clear `RuntimeError`.
- Logging is configured at INFO level, and update-handling exceptions are logged through an aiogram router error handler.
- Repository now includes a root `.gitignore` covering local env files, Python caches, virtual environments, logs, IDE files, and local SQLite databases.
- Minimal teacher-student MVP flow is now present: `/invite` works only for users whose persisted role is `teacher`, generates a one-time invite code, and `/join <code>` lets an unlinked user consume that code once to create a teacher-student link while also adding the teacher and student into the shared `Starter Content` workspace membership used by the current Telegram learning flows.
- A `teacher` can also consume their own invite code for self-testing/self-learning, creating a self-link while preserving the `teacher` role.
- Teachers now also have one minimal homework command: `/assign <student_user_id> <learning_item_id,learning_item_id,...>` creates a fixed assignment for one specific student inside one workspace, stores `assignments.workspace_id`, requires the teacher to have `teacher` membership in that content workspace, requires the target student to be a member of that workspace, rejects mixed-workspace item lists, and immediately sends the student `Вам назначено новое задание` with a `Домашка` button.
- Teachers can now use the same `/assign` command with a topic name as the primary UX path: `/assign <student_user_id> <topic_name>` resolves the topic inside the current shared workspace context, stores the topic title as the human-readable assignment title, and still keeps the legacy raw-id path as fallback.
- Topic access for self-learning is now modeled separately from homework: `student_topic_access` stores workspace-scoped topic access with explicit `workspace_id`, `/granttopic <student_user_id> <topic_name>` grants one topic by name inside the current shared workspace context, `/topics` lists only accessible topics from workspaces where the student is still a member, and starting a topic uses only that topic's linked `learning_item` ids while reusing the existing training session flow.
- Workspace foundation persistence is now present but intentionally narrow: `workspaces` and `workspace_members` store shared ownership membership with only `teacher` and `student` roles, and `englishbot/workspaces.py` provides focused business logic to create a workspace, add/update a member, list a user's workspaces, and find a user's workspaces by role.
- Workspace-owned content MVP is now present: `learning_items` and `topics` carry `workspace_id`, helper lookups default to the compatibility `Starter Content` workspace when no explicit workspace is provided, and `englishbot/topics.py` rejects `topic_learning_items` links that cross workspace boundaries.
- Student homework lists now show the stored assignment title instead of only anonymous `Задание #<id>` entries, and homework start messages include that title.
- Teacher-student links remain the invite/onboarding bridge in the current bot, but workspace membership is now the source of truth for assignment ownership, topic access, homework visibility, and topic visibility in Telegram flows.
- Teacher-student business rules live in `englishbot/teacher_student.py`; workspace foundation business rules live in `englishbot/workspaces.py`; homework business rules live in `englishbot/homework.py`; topic-access business rules live in `englishbot/topic_access.py`; profile persistence lives in `englishbot/user_profiles.py`; thin Telegram command wrappers live in `englishbot/teacher_handlers.py`, `englishbot/homework_handlers.py`, and `englishbot/topic_access_handlers.py`; `englishbot/bot.py` only imports those handlers for registration and extends `/me`.
- Repository now includes a small report utility: `export_ai_file_report.py` writes a text file with each discovered `AGENTS.md` or agent-related file name, relative path, full path, size, and full file contents; discovery logic lives in `englishbot/ai_file_report.py`.
- Focused tests now live in `tests/test_user_profiles.py`, `tests/test_teacher_student.py`, `tests/test_teacher_handlers.py`, `tests/test_vocabulary.py`, `tests/test_training.py`, `tests/test_training_handlers.py`, `tests/test_homework.py`, `tests/test_homework_handlers.py`, `tests/test_basic_topics_seed.py`, `tests/test_topics.py`, `tests/test_topic_access.py`, `tests/test_topic_access_handlers.py`, and `tests/test_workspaces.py`.
- Current constraints stay intentionally narrow: no FSM, no deep links, no middleware changes, no word-level access control, no topic revocation history, no expiry windows, no answer history persistence, no statistics beyond per-session counters, no asset playback, only one minimal fixed-set homework type over assigned `learning_item` ids whether assigned directly or via persisted topics, one separate topic-level access model for self-learning, no workspace UI or workspace-selection flow yet, no sharing/publishing of content between workspaces, and current teacher `/assign` plus `/granttopic` still rely on one implicit shared workspace context ordered by the lowest shared `workspace_id`.
- Focused tests now also cover training service behavior and thin `/learn` command flow without depending on Telegram runtime.
- Changed files for the current slice: `englishbot/db.py`, `englishbot/workspaces.py`, `englishbot/teacher_student.py`, `englishbot/homework.py`, `englishbot/topic_access.py`, `englishbot/teacher_handlers.py`, `tests/test_workspaces.py`, `tests/test_teacher_student.py`, `tests/test_homework.py`, `tests/test_teacher_handlers.py`, `tests/test_topic_access.py`, `CHANGELOG.md`, and `context/current_state.md`.
