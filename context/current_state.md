# Current State

- Repository currently contains a minimal runnable Telegram bot foundation at `englishbot/`.
- Entry point is `python -m englishbot`.
- Runtime uses long polling with `aiogram 3.x`.
- Centralized bot-facing i18n is now present in `englishbot/i18n.py`: supported UI languages are `en`, `ru`, `uk`, and `bg`; the explicit default/fallback language is `en`; touched Telegram replies, usage messages, button labels, settings texts, and workbook summaries now resolve through i18n keys instead of local hardcoded strings.
- User profiles now persist per-user bot language in SQLite via `user_profiles.bot_language`; `englishbot/user_profiles.py` exposes focused getters/setters for role plus bot language, invalid or missing values fall back to `en`, and future bot-facing text can resolve against that stored preference.
- A minimal `/settings` flow is now present through `englishbot/settings_handlers.py` and `englishbot/command_registry.py`: the command is registered centrally, shows a one-button settings screen, opens an inline language picker for `en`/`ru`/`uk`/`bg`, persists the chosen bot language, and confirms the change in the newly selected language.
- `/start` is now homework-first for students: it upserts the Telegram user into SQLite, shows a localized main homework button when the student has active assignments, and otherwise replies with a localized no-homework message while keeping `/learn` as a fallback.
- Plain text messages are echoed back through the centralized i18n layer as a localized `You wrote: <text>`-style reply and are tracked only through the low-level `interactions` audit log.
- `/me` shows the user's first name or username, the `telegram_user_id`, and the number of saved text messages.
- `/me` also shows the user's stored `role`.
- Telegram command metadata is now centralized in `englishbot/command_registry.py`: each command has one canonical definition with `name`, i18n-backed `description_key`, `scope`, and `register_after_startup`, handler decorators reuse those canonical names where practical, and startup now registers `/start`, `/learn`, `/me`, and `/settings` through Telegram `setMyCommands` in the default language.
- Telegram command metadata remains centralized in `englishbot/command_registry.py`, but command descriptions are now stored as i18n keys and resolved through the centralized i18n layer at registration time; startup now also registers `/settings` in the default command menu.
- Project instructions in `AGENTS.md` now explicitly require any future Telegram command addition or change to go through `englishbot/command_registry.py` first instead of introducing scattered command literals or parallel registries.
- Project instructions in `AGENTS.md` now also explicitly require all future bot-facing text to go through the centralized i18n layer instead of introducing new hardcoded user-visible strings in Telegram modules.
- Configuration is read from `TELEGRAM_BOT_TOKEN`, with `.env` support via `python-dotenv`.
- SQLite storage uses the stdlib `sqlite3` module with a local `englishbot.sqlite3` file by default, overridable through `ENGLISHBOT_DB_PATH`.
- `requirements.txt` stays minimal but now includes both runtime packages (`aiogram`, `python-dotenv`) and the test dependency `pytest`; SQLite is not listed because it comes from the Python standard library.
- SQLite schema is created automatically on startup and now bootstraps `users`, `user_profiles`, `interactions`, `workspaces`, `workspace_members`, `invites`, `teacher_student_links`, `lexemes`, `learning_items`, `learning_item_translations`, `topics`, `topic_learning_items`, `student_topic_access`, `training_sessions`, `training_session_items`, `assignments`, and `assignment_items`; startup also creates a deterministic default authoring workspace named `Starter Content`, and migrations now backfill `workspaces.kind`, archive flags, topic timestamps, `assignments.workspace_id`, `student_topic_access.workspace_id`, session snapshot columns, and new workbook keys for legacy `topics` and `learning_items`, while SQLite insert triggers fill missing workbook keys for new rows.
- Product-level user data is now separated from Telegram transport data: `users` stores Telegram identity fields, while `user_profiles` stores runtime profile fields such as `role`.
- `interactions` is the low-level audit table for Telegram traffic and stores `telegram_user_id`, `direction` (`in` or `out`), `interaction_type`, `content`, and `created_at`.
- Minimal vocabulary persistence is now present in SQLite: `lexemes` stays global, `learning_items` stores the main learning unit linked to a lexeme and belongs to a workspace through `workspace_id`, now also carries a stable workspace-scoped `workbook_key`, published student copies track `source_learning_item_id`, archived items stay readable by id for historical flows, and `learning_item_translations` stores multiple translations per learning item by language without duplicating workspace ownership.
- Repository now includes a minimal SQLite seeder for starter vocabulary packs; the current seeded packs are `weekdays` (7 items), `months` (6 items), and `colors` (6 items), and startup seeds them into the default `Starter Content` workspace as real persisted `topics` linked to workspace-owned `learning_items` with `ru`, `uk`, and `bg` translations.
- `learning_items` now supports nullable local asset references through `image_ref` and `audio_ref`, with a small startup migration that adds those columns if an older `learning_items` table already exists without them.
- Minimal learner training is now present: `/learn` still creates one active training session per user over a fixed ordered slice of up to 5 `learning_items`, but the fallback compatibility path now reads from the default content workspace instead of treating all learning items as globally usable; it asks one prompt at a time, validates plain-text answers against the lexeme headword, and ends with a short total/correct summary.
- The active-training plain-text answer handler now explicitly ignores slash commands at the router-filter level, so `/me`, `/learn`, and other commands still work while a training session is open instead of being swallowed by the generic text flow.
- Training persistence now uses `training_sessions` for active/completed session state and `training_session_items` for the ordered `learning_item` ids inside a session; `training_sessions.assignment_id` is nullable so the same session flow can be reused for homework, and `training_session_items` now snapshots `prompt_text` plus `expected_answer` so active sessions stay stable after content edits.
- Training prompt selection is intentionally narrow: prefer a `ru` translation, then `uk`, then the first stored translation, and finally fall back to the lexeme headword when no translation exists.
- A standalone exercise domain model now exists in `englishbot/exercises.py`: it can build concrete `easy`, `medium`, and `hard` exercise instances from resolved `learning_item` content, produces strict typed payloads for `multiple_choice`, `jumbled_letters`, and `typed_answer`, enforces the shared hint fallback order (`requested -> ru -> uk -> first available -> headword`), and raises a domain error when `easy` exercises lack enough distractors.
- The current Telegram/runtime exercise flow is still a single plain-text question/answer model; the new exercise builder is not yet wired into `/learn`, homework, or handlers, there is still no persisted exercise-instance or training-stage model, and there is still no separate persisted `hint_language` setting.
- The target specification in `context/englishbot_handoff.md` now includes a future multi-stage exercise model: it separates `learning_item` content from concrete exercise instances and training-session snapshots, defines `easy`/`medium`/`hard` stages plus `multiple_choice`/`jumbled_letters`/`typed_answer` exercise types, introduces completion/unlock rules, separates `bot_language` from future `hint_language`, and notes that Telegram progress/question message replacement is runtime transport behavior rather than core domain state.
- SQLite connections now enable foreign key enforcement, so `learning_items.lexeme_id` and `learning_item_translations.learning_item_id` are validated at runtime instead of acting as soft references.
- Vocabulary create/read persistence lives in `englishbot/vocabulary.py`; it intentionally exposes only the minimal helpers needed to create lexemes, learning items, translations, and read one learning item together with its translations.
- Homework business logic now lives in `englishbot/homework.py`; topic persistence/read helpers live in `englishbot/topics.py`; thin student homework Telegram wrappers live in `englishbot/homework_handlers.py`; `englishbot/training.py` is reused for both `/learn` and homework sessions; and `englishbot/bot.py` still only wires handlers plus `/me`, echo, and startup.
- `/me` counts saved user text messages by counting `interactions` rows where `direction='in'` and `interaction_type='text'`.
- Startup removes the old `messages` table if it exists, because it is no longer used.
- Interaction logging is attached at the aiogram framework level instead of inside specific handlers: incoming updates go through a dispatcher update middleware, and outgoing Bot API requests go through session middleware.
- Audit logging now tolerates first-contact users: before inserting into `interactions`, persistence creates a minimal `users` row plus default `user_profiles` row when that Telegram id has not been saved yet, preventing foreign-key failures on fresh updates.
- If the bot token is missing, startup fails with a clear `RuntimeError`.
- Logging is configured at INFO level, and update-handling exceptions are logged through an aiogram router error handler.
- Repository now includes a root `.gitignore` covering local env files, Python caches, virtual environments, logs, IDE files, and local SQLite databases.
- Minimal teacher-student MVP flow is now present: `/invite` works only for users whose persisted role is `teacher`, generates a one-time invite code, and `/join <code>` lets an unlinked user consume that code once to create a teacher-student link while ensuring the teacher has access to the default teacher workspace and creating one shared `student` workspace for runtime learning flows.
- A `teacher` can also consume their own invite code for self-testing/self-learning, creating a self-link while preserving the `teacher` role.
- Teachers now also have one minimal homework command: `/assign <student_user_id> <learning_item_id,learning_item_id,...>` creates a fixed assignment for one specific student inside that teacher-student pair's shared `student` workspace, stores `assignments.workspace_id`, auto-publishes teacher-workspace learning items into the student workspace when needed, rejects mixed source workspaces, and immediately sends the student `Вам назначено новое задание` with a `Домашка` button.
- Teachers can now use the same `/assign` command with a topic name as the primary UX path, but topic-based assignment is now explicit and workspace-scoped: `/assign <student_user_id> <teacher_workspace_id> <topic_name>` resolves the topic only inside the specified teacher workspace after validating teacher membership there, publishes that topic into the shared `student` workspace, stores the topic title as the human-readable assignment title, and keeps the raw-id `/assign <student_user_id> <learning_item_id,learning_item_id,...>` path unchanged.
- Topic access for self-learning is now modeled separately from homework: `student_topic_access` stores workspace-scoped topic access with explicit `workspace_id`, `/granttopic <student_user_id> <teacher_workspace_id> <topic_name>` publishes only the specified teacher-workspace topic into the shared `student` workspace before granting access, `/topics` lists only accessible non-archived topics from student workspaces where the learner is still a member, and starting a topic uses the published topic's current linked `learning_item` ids while reusing the existing training session flow.
- Workspace foundation persistence is now present but intentionally narrow: `workspaces` and `workspace_members` store explicit workspace `kind` (`teacher` or `student`) plus member roles (`teacher` or `student`), and `englishbot/workspaces.py` provides focused business logic to create a workspace, add/update a member, list a user's workspaces, create deterministic teacher-student student workspaces, and enforce teacher-only editing of teacher workspaces; shared student workspace lookup now returns a workspace only when exactly one matching student workspace exists and otherwise fails closed instead of picking a first row.
- Workspace-owned content MVP is now present: `learning_items` and `topics` carry `workspace_id`, both now also carry stable workspace-scoped `workbook_key` values, published student copies still track `source_learning_item_id` or `source_topic_id` but keep their own local workbook keys, teacher-only mutation helpers now cover create/update/archive flows, list/discovery helpers hide archived rows by default, and `englishbot/topics.py` still rejects `topic_learning_items` links that cross workspace boundaries.
- Confirmed persistence mappings for the future workbook slice are now documented from real code instead of assumptions: topics use `name` plus `title`; learning items use `text`, `image_ref`, and `audio_ref`; headwords live in `lexemes.lemma`; translations live in `learning_item_translations(language_code, translation_text)`; topic membership lives in `topic_learning_items`; and current topic ordering is implicit by relation-row insertion order rather than a stored `sort_order`.
- Minimal workbook tooling is now present for teacher workspaces only: `englishbot/workbook_export.py` exports one authorized teacher workspace to an in-memory `.xlsx` containing exactly `topics`, `topic_items`, and `learning_items` sheets in deterministic order, while `englishbot/workbook_import.py` performs one-workbook-into-one-teacher-workspace atomic partial upserts by `workbook_key`; workbook keys remain the primary identity, `ru`/`uk`/`bg` translations are wide, `topic_items` links are additive, `image_ref` is the source of truth, the display-only `image` column is ignored on import, and import validation now fails fast on missing sheets/columns, duplicate workbook keys, broken refs, and invalid archive values.
- Minimal Telegram workbook admin flows are now present for teacher workspaces only: `/workbook_export <teacher_workspace_id>` returns the workspace workbook as a Telegram `.xlsx` document with a short content summary, and workbook import works by uploading a `.xlsx` file with caption `/workbook_import <teacher_workspace_id>`; both flows require explicit teacher workspace ids, reject student workspaces clearly, enforce teacher membership in the target teacher workspace, and keep the Telegram layer thin by reusing `englishbot/workbook_export.py`, `englishbot/workbook_import.py`, and `englishbot/workbook_admin.py`.
- Student homework lists now show the stored assignment title instead of only anonymous `Задание #<id>` entries, and homework start messages include that title.
- Teacher-student links remain the invite/onboarding bridge in the current bot, but workspace membership is now the source of truth for assignment ownership, topic access, homework visibility, and topic visibility in Telegram flows, with published student workspaces acting as the runtime target instead of the teacher catalog.
- Teacher-student business rules live in `englishbot/teacher_student.py`; workspace foundation business rules live in `englishbot/workspaces.py`; homework business rules live in `englishbot/homework.py`; topic-access business rules live in `englishbot/topic_access.py`; workbook admin orchestration lives in `englishbot/workbook_admin.py`; profile persistence lives in `englishbot/user_profiles.py`; thin Telegram command wrappers live in `englishbot/teacher_handlers.py`, `englishbot/homework_handlers.py`, `englishbot/topic_access_handlers.py`, and `englishbot/workbook_handlers.py`; `englishbot/bot.py` only imports those handlers for registration and extends `/me`.
- Repository now includes a small report utility: `export_ai_file_report.py` writes a text file with each discovered `AGENTS.md` or agent-related file name, relative path, full path, size, and full file contents; discovery logic lives in `englishbot/ai_file_report.py`.
- Focused tests now live in `tests/test_bot.py`, `tests/test_command_registry.py`, `tests/test_exercises.py`, `tests/test_user_profiles.py`, `tests/test_teacher_student.py`, `tests/test_teacher_handlers.py`, `tests/test_vocabulary.py`, `tests/test_training.py`, `tests/test_training_handlers.py`, `tests/test_homework.py`, `tests/test_homework_handlers.py`, `tests/test_basic_topics_seed.py`, `tests/test_topics.py`, `tests/test_topic_access.py`, `tests/test_topic_access_handlers.py`, `tests/test_workbook_admin.py`, `tests/test_workbook_handlers.py`, `tests/test_workbook_export.py`, `tests/test_workbook_import.py`, and `tests/test_workspaces.py`; the command-registry suite now covers centralized command discovery, registration-flag filtering, and canonical command-name reuse in usage/caption parsing, while the bot suite still covers `/me` output, startup command registration, and the regression where `/me` must still answer during an active training session.
- Current constraints stay intentionally narrow: no FSM, no deep links, no middleware changes, no word-level access control, no topic revocation history, no expiry windows, no answer history persistence, no statistics beyond per-session counters, no asset playback, only one minimal fixed-set homework type over assigned `learning_item` ids whether assigned directly or via persisted topics, one separate topic-level access model for self-learning, no workspace UI or workspace-selection flow yet, workbook support is still local-file based with no Google Sheets API/versioning/diff workflow, no diff-based publishing, no content versioning, no back-sync from student to teacher workspaces, and teacher topic commands still require an explicit teacher workspace id instead of any implicit authoring-workspace fallback.
- `context/englishbot_handoff.md` is now maintained as a merged technical specification instead of a target-only note: it preserves future design sections, but now explicitly records current workbook-key identity, workbook import/export semantics, workspace behavior, publish boundaries, assignment/training snapshot behavior, terminology, and current platform constraints in a way that matches this file.
- Focused tests now also cover training service behavior and thin `/learn` command flow without depending on Telegram runtime.
- Focused tests now also cover centralized i18n lookup/fallback plus the minimal `/settings` language-selection flow in `tests/test_i18n.py` and `tests/test_settings_handlers.py`, alongside updated persistence/handler assertions for the new default-language behavior.
- Changed files for the current slice: `englishbot/exercises.py`, `tests/test_exercises.py`, `CHANGELOG.md`, and `context/current_state.md`.
