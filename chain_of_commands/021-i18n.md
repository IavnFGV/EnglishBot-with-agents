Goal:
Introduce centralized internationalization (i18n) for all bot-facing text interactions and allow each user to choose their preferred bot language.

Context:
The bot now needs a single, explicit internationalization layer for all user-visible text.
This must cover current and future Telegram interactions.

Supported bot languages in this slice:
- English (`en`)
- Russian (`ru`)
- Ukrainian (`uk`)
- Bulgarian (`bg`)

Target model:
- all bot-facing text must be resolved through a centralized i18n layer
- each user must have a stored preferred bot language
- language selection must be available through a minimal in-bot settings flow
- future features must use the same i18n layer instead of hardcoded text

Feature:
Centralized Bot Internationalization + User Language Preference (MVP)

Scope:

1. Introduce one focused i18n module or package
   - all bot-facing text keys must be resolved through it
   - avoid scattered hardcoded user-visible strings in handlers
   - keep the solution explicit and understandable

2. Supported languages:
   - en
   - ru
   - uk
   - bg

3. Add persistent user language preference
   - store per-user preferred bot language in persistence
   - define a clear default language for users who have not chosen yet

4. Introduce minimal settings flow in Telegram
   - add `/settings`
   - provide a language selection entry inside settings
   - allow user to choose one of the supported bot languages
   - after selection, future bot responses should use that language

5. Centralize text resolution
   - existing bot-facing responses should be moved to i18n keys where practical in this slice
   - new bot-facing text introduced by this slice must use i18n only
   - do not leave newly added hardcoded user-visible strings in handlers

6. Command/help/menu texts
   - if settings/language-related command descriptions are registered centrally, they must also follow the i18n model where practical
   - keep Telegram command registration behavior simple in this slice

7. Sub-agent instructions / project guidance update
   - update project guidance files so future implementation work does not introduce new hardcoded bot-facing strings
   - make it explicit that all bot-facing textual interactions must go through the i18n layer

Cleanup rules:
- remove replaced hardcoded bot-facing strings where practical
- do not keep parallel ad-hoc text-definition styles without justification
- do not bury translation dictionaries inside handler modules
- do not leave new settings/language texts outside the i18n layer

Out of scope:
- full pluralization framework unless minimally needed
- locale-specific formatting overhaul
- per-workspace language
- content translation management for learning content itself
- admin UI for translation editing
- dynamic external translation backends

Architecture constraints:
- keep implementation minimal and centralized
- prefer one focused i18n layer over scattered maps in many modules
- do not introduce a heavy framework
- keep business logic outside Telegram handlers
- handlers may request resolved text, but should not become text stores

Important behavioral requirement:
- all user-visible Telegram interaction text must be considered bot-facing text
- this includes:
  - command replies
  - validation errors shown to user
  - menu labels
  - inline button labels
  - settings texts
  - success/failure user messages
- internal logs and developer-only diagnostics do not need i18n in this slice

Persistence requirement:
- user preferred bot language must be stored persistently
- default language behavior must be explicit and deterministic

Testing requirements:
- tests for language preference persistence
- tests for i18n lookup behavior
- tests for fallback/default language behavior
- tests for settings language change flow
- tests that key bot-facing texts in the touched flows are resolved through i18n
- focused tests without Telegram runtime where possible

Documentation:
- update CHANGELOG.md
- update context/current_state.md
- update AGENTS.md and any relevant agent/project guidance so future changes must use the i18n layer for bot-facing text
